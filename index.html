Session、Token、JWT、OAuth2 的区别

首先我对于几个概念进行阐述和区分

从分类上来说

  • Session： 是一种认证机制
  • Token、JWT： 是认证授权机制
  • OAuth2： 是授权框架

从应用场景上来说

  • Session： 自己的简单网站
  • Token： 分布式、跨系统、单点登录、允许第三方调用 api 接口
  • JWT： 场景可以同 Token 一致，但是获取验证 Token 的实现逻辑不一样
  • OAuth2： 授权第三方应用、多平台单点登录

从具体实现来说

  • Session： 基于 cookie 和 sessionId
  • Token：uid（用户唯一身份标识） + timestamp（当前时间戳）
  • JWT：header（头部） + payload（负载） + sign（签名）
  • OAuth2：基于授权码 code、client_id、client_secret

JWT

什么是 JWT

  • JSON Web Token （简称 JWT）是目前最流行的跨域认证解决方案
  • JWT 是为了在网络应用环境间传递声明而执行的一种基于 JSON 的开放标准（RFC 7519）。JWT 的声明一般被用来在身份提供者和服务提供者间传递被认证的用户身份信息，以便于从资源服务器获取资源
  • 可以使用 HMAC 算法或者是 RSA 的公/私秘钥对 JWT 进行签名。因为数字签名的存在，这些传递的信息是可信的

WT （Web Token）

WT包含三个部分：Header头部，Payload负载和Signature签名，三个部分之间用 “.” 做分割

  1. Header：在其中通常包含了两个部分：type：代表 token 的类型，alg：使用的 Hash 算法，例如 HMAC SHA256 或 RSA
    如：{“alg”: “HS256”, “type”: “JWT”}

  2. Payload token：它包含一些声明 Claim 声明分三类：

  • Reserved Claims：这是一套预定义的声明这不是必须的，这是一套易于使用、操作性强的声明，如：iss(issuer)、exp(expiration time)、sub(subject)、aud(audience)等
  • Plubic Claims
  • Private Claims：交换信息的双方自定义的声明，经过 Base64Url 编码后形成第二部分，如：{“sub”: “1234567890”, “name”: “Fall”, “admin”, “true”}
  1. signature：使用 header 中指定的算法将编码后的 header、编码后的 payload、一个 secret 进行加密。如：HMACSH256(base64UrlEncode(header) + “.” + base64UrlEncode(payload), secret) 这个 signature 字段被用来确认 JWT 信息的发送者是谁，并保证信息没有被修改

JWT 认证流程

  • 用户输入用户名/密码登录，服务端认证成功后，会返回给客户端一个 JWT
  • 客户端将 token 保存到本地
  • 当用户希望访问一个受保护的路由或者资源的时候，需要请求头的 Authorization 字段中使用 Bearer 模式添加 JWT，如 Authorization: Bearer + token

OAuth

在 OAuth 中 O 是 Open 的简称，表示开放的意思。Auth 表示 “授权” 的意思，所以它的意思是 “开发授权” 的意思，其是一个关于授权（authorization）的开放网络标准，总的来说其是一种授权协议

其允许用户授权第三方应用访问他在存储在另外服务商里的各种信息数据，而这种授权不需要提供用户名和密码提供给第三方应用，这种授权协议，就是保证第三方应用只有在获得授权之后，才能进一步访问授权者的数据

OAuth2 授权模式

  • 授权码模式
  • 简化模式
  • 密码模式
  • 客户端模式

线程、进程、并行、并发的概念

发表于 2023-08-17 | 更新于: 2023-08-17 | 分类于 多线程

并发 和 并行

概念

  • 并发：在单核 CPU 中，系统调度在某一时刻只能让一个进程运行，虽然这种调度机制有多种形式，只要通过不断切换需要运行的进程让其运行的方式叫并发

  • 并行：在多核 CPU 中，可以让两个以上的进程同时运行在不同的物理核心上，这种运行方式就是并行

区别

  • 并发在微观上不是同时执行的，只是把时间分成若干段，使多个进程快速交替的执行

  • 并行是真正在同时进行的，即同一时间点上同时运行者多个进程

进程 和 线程

概念

  • 进程：在计算机中的程序关于某数据集合上的一次运行活动，是系统进行资源分配的基本单位，在早期面向进程设计的计算机结果中，进程是程序的基本执行实体，在当代面向线程设计的计算机结果中，进程是线程的容器

  • 线程：操作系统能够运算调度的最小单位。它被包含在进程中，是进程中的实际运作单位。一条线程指的是进程中一个单一顺序的控制流，一个进程中可以并发多个线程，每个线程并行执行不同任务。线程是独立调度和分派的基本单位。

线程状态流转图

区别

进程和线程的主要区别在于他们是不同的操作系统资源管理方式。进程具有独立的地址空间，一个进程崩溃后，在保护模式下不会对其他进程产生影响，而线程只是一个进程中不同执行路径。线程有自己的堆栈和局部变量，但线程之间没有单独的地址空间，一个线程死掉等于整个进程死掉，所以多进程程序比多线程程序更加健壮，但在进程切换时，消耗资源较大，效率低。

对于一些要求同时进行并且又要共享某些变量的并发操作，只能用线程，不能用进程

总结

  • 一个程序至少有一个进程，一个进程至少有一个线程
  • 线程的划分尺度小于进程，使得线程的并发性高
  • 进程在执行过程中拥有独立的内存单元，而线程共享内存，这极大的提高了程序的运行效率
  • 每个独立的进程有个程序运行的入口、顺序执行序列和程序的出口。但是线程不能独立执行，必须依存于进程中
  • 多线程的意义在于一个进程中，有多个执行部分可以同时执行。但操作系统并没有将多个线程看做多个独立的应用，来实现进程的调度和管理以及资源分配

锁

为什么需要锁

在多线程并发场景下，为了防止多个线程同时访问同一个内存地址，造成数据的不一致，所以引入了锁的概念

锁的分类

互斥锁

保证同一时间段，只有一个线程可以对临界资源进行操作，也就是说当线 A 在操作临界资源时，线程 B 无法操作其，只有当线程 A 退出临界资源线程 B 才能对临界资源操作

本质上来说互斥锁是通过一个 state 变量，来标记锁的状态，对 state 操作来保证线程的安全

公平锁

指多线程按照顺序申请锁获取锁，线程直接进入队列中进行排队，只有队列中的第一个线程可以获得锁

非公平锁

多个线程在加锁时，会直接抢锁，只有抢不到才会进入队列中进行排序

这两个锁的实现都是由 ReentrantLock 中 Sync 的内部类实现的，其实继承的是 AbstractQueuedSynchronizer，该抽象类本质通过 state 和自身实现的队列，来完成互斥锁中的操作，即当 state 中有值的时候代表，已经有线程获取该锁了，需要经行公平或非公平抢锁。同时又因为 state 本身是一个 int 类型的变量也可以通过 state 的值判断一共有几个正在等待的线程

自旋锁

其底层是基于 CAS (compare and swap) 这样的机制去解决线程安全的问题，其会不断地进行比较，也就是用一个旧值的预期值和内存值比较，如果两个值相等，就用新的值替换内存值，否则 while 循环直到操作完成

顺序锁

当读者在读的时候也允许写者继续运行，这样写者就不用再等待了，缺点是是有些时候读者不得不反复多次相同的数据直到它获得有效的副本

乐观锁

指令重排序问题

发表于 2023-08-05 | 更新于: 2023-08-05 | 分类于 多线程

为什么需要重排序

一个简单的问题 CPU 的资源的优化，当 CPU 在执行任务 A 时，同时可以执行任务 B, 因为任务 A B 之间没有任何依赖关系，所以任务 A B 重排，提高了 CPU 的效率

为什么会产生问题

首先明确一点，重排序是必须的，因为在计算机系统中 CPU 是极其宝贵的资源。

那么问题是如何产生的呢，一般在 CPU (X86) 中每个核心拥有 L1、L2缓存独占和公用的 L3 缓存，在 CPU 中的缓存拥有一致性协议，其缓存不会出现不一致的问题，也就不会有内存可见性的问题
虽然这几个缓存是有一致性协议，但是协议吗，对于 CPU 的性能是由损耗的，所以在 CPU 内部还有 Store Buffer、Load Buffer 等之类的 Buffer，这些 Buffer 与其他的缓存是不一致的，是异步的。

也就可以说 CPU 和主内存是异步，就有内存可见性的问题。

举个例子也就是会产生如下问题：

其会产生那么几个结果:

  • a=0, b=1
  • a=1, b=0
  • a=1, b=1

这就是由于内存可见性和重排序共同产生的问题

重排序场景

编译器重排序

对于没有先后依赖关系的语句，编译器可以重新调整语句的执行顺序

CPU 指令重排序

在指令级别，让没有依赖关系的多条指令并行

CPU 内存重排序

CPU 有自己的缓存，指令的执行顺序和写入主内存的顺序不完全一致

as-if-serial 语义

既然我们已知晓，其产生的问题和场景，那么我们现在就要开始解决这么个问题了

单线程

解决多线程的问题前，我们先论述单线程的解决方案 **as-if-serial 语义**，只要是单线程无论其是什么语言，只要操作之间没有数据依赖性，那么如何执行都是没有问题的，代码看起来就像完全串行地一行行执行一遍。

多线程

那么在多线程中其是如何的呢，由于多线程的复杂性，CPU 和编译器无法完全理解，从而给出最合理的优化。

所以在多线程的解决方案就是，保证每个线程内部都是 **as-if-serial 语义**，但对于公共的变量和线程之间的读写，则不会考虑

happen-before

为了解决在多线程场景下，什么时候可以重排序，什么时候不可以进行重排序，Java 引入 JMM 这个模型来定义一套规范，然后通过 happen-before 去描述这个规范。

那么 happen-before 到底是什么呢，假设 A happen-before B，那么就代表如果 A 在 B 之前执行，则 A 的执行结果必须对 B 可见。定义了可见性的约束，也就定义了重排序的约束。

同时 happen-before 具有传递性，即如 A happen-before B， B happen-before C，就有 A happen-before C 这个性质。

happen-before 规则

  • 单线程中的每个操作，happen-before 于该线程中任意后续操作
  • 对 volatile 变量的写，happen-before 于后续这个变量的读
  • 对 synchronized 的解锁，happen-before 于后续这个锁的加锁
  • 对 final 变量的写，happen-before 于 final 域对象的读，happen-before 于后续对 final 变量的读

内存屏障

当然以上都是为了解决 CPU 内存重排序这个场景的问题，那么的剩下的问题如何解决呢，这时候就引入了内存屏障的概念了。

编译器的内存屏障，只是为了告诉编译器不要对那些指令进行重排序。当编译完成，之后这种屏障就会消失了，CPU 不会感知到他的存在。

CPU 的内存屏障是 CPU 提供的指令，可以由开发者直接调用

CPU 内存屏障

  • LoadLoad: 禁止读和读的重排序
  • StoreStore: 禁止写和写的重排序
  • LoadStore: 禁止读和写的重排序
  • StoreLoad: 禁止写和读的重排序

JDK 内存屏障

JDK 中的内存屏障是由 CPU 内存屏障组合而成

  • loadFence = LoadLoad + LoadStore
  • storeFence = StoreStore + LoadStore
  • fullFence = loadFence + storeFence + StoreLoad

消费者-生产者 模型

定义

在生产者／消费者模型中，生产者Producer负责生产数据，而消费者Consumer负责使用数据。多个生产者线程会在同一时间运行，生产数据，并放到内存中一个共享的区域

优势

  • 解耦

    将生产者类和消费者类进行解耦，消除代码之间的依赖性，简化工作负载的管理。

  • 支持分布式

    生产者和消费者通过队列进行通讯，所以不需要运行在同一台机器上，在分布式环境中可以通过 redis 的 list 作为队列，而消费者只需要轮询队列中是否有数据。同时还能支持集群的伸缩性，当某台机器宕掉的时候，不会导致整个集群宕掉。

  • 复用

    通过将生产者类和消费者类独立开来，那么可以对生产者类和消费者类进行独立的复用与扩展。

  • 调整并发数

    由于生产者和消费者的处理速度是不一样的，可以调整并发数，给予慢的一方多的并发数，来提高任务的处理速度。

  • 异步

    对于生产者和消费者来说能够各司其职，生产者只需要关心缓冲区是否还有数据，不需要等待消费者处理完；同样的对于消费者来说，也只需要关注缓冲区的内容，不需要关注生产者，通过异步的方式支持高并发，将一个耗时的流程拆成生产和消费两个阶段，这样生产者因为执行 put() 的时间比较短，而支持高并发。

心跳机制实现

心跳机制原理

客户端或者说是服务提供者(需要被注册的服务)需要每隔一段时间向服务端或者服务发现者发送一个数据包, 同时在服务端接收到服务包之后也会返回一个数据包相应客户端, 如果双方在一定时间内没有接收到对方的数据表, 则断开连接

心跳机制技术

  1. TCP

    在 TCP 的机制中本身就带有心跳机制, 只要一方开启 KeepAlive 功能以后, 就会自动在规定时间内向对方发送数据包

  2. 应用层自定义心跳包

    客户端启动一个线程定时向服务端发送数据表, 同时服务端启动一个线程维护客户端发送数据包的时间是否出现异常
